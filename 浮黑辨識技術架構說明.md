## 1. 系統總覽

### 1.1 功能定位

本系統將「水泥浮黑目視估值」自動化：使用者用手機正上方俯拍試體照片，
系統回傳「浮黑物質佔試體表面的百分比（0–100%）」，並把所有量測過程入庫，供日後品質追溯與校準回饋。

### 1.2 高階流程

```
┌────────────────────────────────────────────────────┐
│  使用者拍照（手機）                                │
└────────────────┬───────────────────────────────────┘
                 ▼
┌────────────────────────────────────────────────────┐
│  Streamlit 介面（瀏覽器訪問）                      │

└────────────────┬───────────────────────────────────┘
                 ▼
┌────────────────────────────────────────────────────┐
│  cement_core.detector.predict()                    │
│  ├─ 1. compression（縮小至約 1.5–2 MB）            │
│  ├─ 2. preprocessing（白平衡校正）                 │
│  ├─ 3. roi（HoughCircles 找燒杯）                  │
```

## 2. 部署架構

### 2.1 雙平台並存

```
                    GitHub repo (main branch)
                    ├─────────────┐
                    ▼             ▼
              push 觸發       sync-hf.yml
                    │             │
                    ▼             ▼
        Streamlit Cloud    HuggingFace Space
        (streamlit.app)    (hf.space)
                    │             │
                    └─────────────┘
                          │
                          ▼
                       Supabase
                  (DB + Storage 共用)
```

**設計理由**：
- Streamlit Cloud 免費版在低流量時會休眠（首訪冷啟動 30+ 秒）
- HuggingFace Spaces 也會休眠，但門檻為 48 小時無訪問
- 兩平台共用同一個 Supabase 後端，**資料一致，故障時可手動切換訪問入口**
- GitHub Actions `sync-hf.yml` 推送同步 HF Space，避免人工部署兩次

### 2.2 secrets 管理

每個平台都需設置以下 secrets（建議 12+ 字元、含大小寫數字符號）：

```
SUPABASE_URL         = https://<project>.supabase.co
SUPABASE_KEY         = sb_secret_xxx          # 必須是 secret key，非 publishable
SUPABASE_DB_URL      = postgresql://...:6543/postgres   # pooler 連線

ACCOUNT_ADMIN_PW     = <管理員密碼>
ACCOUNT_HEPING_PW    = <和平廠密碼>
ACCOUNT_SUAO_PW      = <蘇澳廠密碼>
# ... 其餘廠別比照
```

**SUPABASE_KEY 必為 `sb_secret_xxx`**：
- 公開金鑰（publishable）受 RLS 限制，無法繞過儲存 Bucket 的 RLS policy
- 應用伺服器側用 secret key 才能無條件操作 Storage

**SUPABASE_DB_URL 必為 pooler 連線（port 6543）**：
- 避免長連線造成 Streamlit 端常斷線
- 注意：transaction-pool 模式下，session 設定（如 SET TIME ZONE）易丟失，故 v16.4 改用 `ALTER DATABASE` database-level 時區

### 2.3 GitHub Actions 同步流程

`.github/workflows/sync-hf.yml`：
```yaml

env:
  HF_USER: <your_hf_username>
```

### 3.4 預測快取機制（防 DB 洗版）

問題：Streamlit 任何 widget 互動觸發 rerun，預測迴圈會重跑。若無快取，每次 rerun 都會新增一筆 DB 紀錄。

設計（streamlit_app.py L208–230）：
1. 上傳檔案後計算 `file_hash = md5(file_bytes)`
2. 用 `cache_key = f"{file_hash}_{test_account}"` 查 session_state
3. 命中 → 跳過 predict，直接用快取
4. 未命中 → 呼叫 detector.predict → 存入快取
5. 「🔄 重新測試此圖」按鈕：手動 pop 該 cache_key → rerun → 重新預測 → DB 新增紀錄

**為什麼 cache key 要含 test_account**：
管理員切換到「和平」身份測試同一張圖時應該被視為「另一次測試」，
不能用之前掛在「管理員」名下的快取結果。

## 4. 核心套件（cement_core）

### 4.1 compression.py

#### 目的
上傳前壓縮影像至約 1.5–2 MB，加速網路傳輸與後續處理。

#### 核心邏輯
- 輸入：PIL Image 或 numpy array
- 輸出：JPEG quality 85、最長邊 1920px 以內
- 同時計算 `file_hash`（壓縮後 bytes 的 MD5）作為內容指紋

#### API
```python
def compress(rgb: np.ndarray) -> Tuple[bytes, float]
def file_hash(compressed_bytes: bytes) -> str
```

### 4.2 preprocessing.py

#### 4.2.1 白紙偵測 — `detect_white_paper`

判定條件：
- HSV space：`V > 200` 且 `S < 30`
- 最大連通區占畫面 ≥ 15%
- 該區延伸到畫面邊緣（距離邊緣 < 10% 影像寬高）
- 排除「畫面中央的小白塊」（避免誤把試體頂端反光當白紙）

設計動機：在自然光下，紙張的 V 通道反光可能與試體高光接近，需以位置與大小條件區分。

#### 4.2.2 白平衡校正 — `apply_white_balance`

演算法：
1. 計算白紙區的 L\* 中位數（CIE L\*a\*b\* 色彩空間，CIE 1976 [3]）
2. 校正因子 `wb_factor = 95 / median_L`（目標 L\* = 95，標準白）
3. 把整張圖 L\* 通道乘上 wb_factor，再轉回 RGB

#### 4.2.3 解析度標準化 — `normalize_resolution`
把 ROI 裁切到固定大小（預設 512×512），讓 rel/abs 量測在不同手機解析度下可比較。

### 4.3 roi.py

#### 演算法
**Hough Circle Transform**（Duda & Hart 1972 [4]）找燒杯圓形：

```python
circles = cv2.HoughCircles(
    blurred_gray,
    cv2.HOUGH_GRADIENT, dp=1,
    minDist=max(50, min_r),
    param1=50, param2=30,
    minRadius=int(min(w, h) * 0.18),
    maxRadius=int(min(w, h) * 0.48),
)
```

#### 結果
回傳 `(center_x, center_y, radius)`，並產生圓形 mask 供量測使用。
失敗時退回 fallback：以畫面中心為圓心、預設半徑（30% 短邊）。

#### 限制
HoughCircles 對「圓形邊緣對比清晰」的場景效果好。若：
- 試體中浮黑物極多 → 邊緣對比降低 → 偵測機率下降
- 強反光蓋住部分圓周 → 偵測機率下降
故偵測失敗也不阻擋預測流程，僅在品質警告中提示。

### 4.4 measurement.py

#### 4.4.1 兩個浮黑指標

**rel\_dark\_pct（相對暗區比）**：
- 以 ROI 內所有像素的中位 L\* 作為「白底基準」
- 計算「L\* < (中位 L\*) × rel_threshold」的像素占比
- 預設 rel_threshold = 0.83（即比基準暗 17% 以上的視為浮黑）

**abs\_dark\_pct（絕對暗度比）**：
- 計算「L\* < abs_threshold」的像素占比
- 預設 abs_threshold = 50（CIE L\* 0–100 範圍）

#### 4.4.2 兩個指標的互補性

| 場景 | rel 表現 | abs 表現 |
|---|---|---|
| 低浮黑（< 20%）| 敏感、線性 | 雜訊大 |
| 中浮黑（20–60%）| 線性 | 線性 |
| 高浮黑（> 80%）| **飽和**（中位 L\* 已被浮黑拉低）| 線性 |

故 v13+ 校準同時擬合兩條 spline（rel/abs 各一條），預測時：
- 一般情況：兩條結果加權平均
- rel 超過校準上界 → 視為飽和，只用 abs

#### 4.4.3 品質評估 — `assess_quality`

四個獨立判定：
| 子項 | 閾值 | 警告訊息 |
|---|---|---|
| 亮度 | 平均 L\* ∈ [60, 200] | 亮度異常(<60 或 >200) |
| 對比 | std(L\*) > 8 | 對比過低（過曝或過暗）|
| ROI 覆蓋 | ROI 占畫面 > 15% | ROI 太小 |
| **反光（v15+）** | `(L\* > 200) 像素佔比 < 1%` | 表面反光（佔比） |

#### 4.4.4 反光偵測（v15 新增）

問題：水光反射會把試體某區域 L\* 拉高到接近白底，影響 rel 中位 L\* 基準。
偵測：
```python
L = rgb_to_lab(image)[:, :, 0]
spec_pixels = L[roi_mask > 0]
specular_ratio = (spec_pixels > 200).mean()
specular_ok = specular_ratio <= 0.01
```

設計動機：避免複雜的去反光演算法（如 dark channel prior, He et al. 2011 [5]），改在 SOP 階段提醒使用者「避免反光」即可。

### 4.5 calibration.py — 核心演算法

#### 4.5.1 Smooth Isotonic Regression（Jiang et al. 2011 [1]）

**問題**：校準資料（一組 (rel\_measured, target\_pct) 點對）必須單調遞增，但因量測雜訊可能違反。

**傳統做法**：強制 `y[i+1] = max(y[i+1], y[i] + epsilon)`。缺點是在量測雜訊處製造**人造拐點**，外推時不穩定。

**Smooth Isotonic Regression 三步驟**：

1. **`IsotonicRegression`**（sklearn）找最佳單調近似
   求解 `min Σ(y_i - ŷ_i)² s.t. ŷ_1 ≤ ŷ_2 ≤ ... ≤ ŷ_n`，使用 Pool Adjacent Violators (PAV) 演算法。
2. **Flat regions 合併**：相鄰相等的 ŷ 值合併為單點（取 x 的平均）。
3. **`PchipInterpolator`**（SciPy，Fritsch & Carlson 1980 [2]）做平滑插值
   保留單調性、避免 over-shoot、C¹ 連續。

```python
def smooth_isotonic_fit(x: np.ndarray, y: np.ndarray) -> PchipInterpolator:
    # Step 1
    iso = IsotonicRegression(out_of_bounds='clip').fit(x_sorted, y_sorted)
    y_iso = iso.predict(x_sorted)
    
    # Step 2
    unique_y, indices = np.unique(y_iso, return_inverse=True)
    x_merged = [np.mean(x_sorted[indices == k]) for k in range(len(unique_y))]
    
    # Step 3
    return PchipInterpolator(x_merged, unique_y)
```

#### 4.5.2 自動最佳閾值搜尋 — `CalibrationOptimizer`

對給定的校準資料，網格搜尋 (rel_threshold, abs_threshold) 組合：
- rel_threshold ∈ [0.75, 0.90]，step 0.01
- abs_threshold ∈ [40, 70]，step 5

選擇能最小化 LOO (Leave-One-Out) MAE 的組合。

#### 4.5.3 LOO MAE 評估

對 n 個校準點：
1. 從資料中拿掉第 i 點
2. 用其餘 n-1 個點建 spline
3. 預測拿掉那點的 rel → 與真值比較
4. 重複 i=1..n，取平均絕對誤差

實測：mi-20/40/50/70/100 五點，LOO MAE ≈ 5–6%。

### 4.6 detector.py — 協調器

#### 4.6.1 calibrate() 流程

```python
def calibrate(self, samples: List[Tuple[Image, float]], name, description):
    for img, target_pct in samples:
        # 1. 壓縮 + 預處理 + ROI
        rgb_c, mask, wb_factor = self._process(img)
        # 2. 量測（純運算，不寫 DB）
        rel, abs_, med = measurement.measure(rgb_c, mask, wb_factor)
        rows.append((rel, abs_, med, target_pct))
    
    # 3. 自動優化 (rel_thr, abs_thr)
    opt = CalibrationOptimizer(rows).optimize()
    self._rel_thr, self._abs_thr = opt.best_thresholds
    
    # 4. 建立兩條 spline
    self._spline_rel = smooth_isotonic_fit(rels, target_pcts)
    self._spline_abs = smooth_isotonic_fit(abss, target_pcts)
    
    # 5. 寫入 calibrations 表（不寫 images/measurements/calibration_samples）
    self.calibration_id = self.db.insert_calibration(
        name=name, description=description,
        rel_thr=self._rel_thr, abs_thr=self._abs_thr,
        rel_anchors=[(r, p) for r, p in zip(rels, target_pcts)],
        abs_anchors=[(a, p) for a, p in zip(abss, target_pcts)],
        loo_mae=loo_mae,
    )
```

**v16.3 之後校準資料僅入 `calibrations` 一張表**（含 rel\_anchors\_json、abs\_anchors\_json），
不再寫 images/measurements/calibration_samples，因為量測值已序列化在 JSON 內。
這簡化了刪除流程、消除孤兒影像問題。

#### 4.6.2 predict() 流程

```python
def predict(self, image_input, label='user_upload', save_to_db=True, account='admin'):
    # 1. 壓縮 + 預處理 + ROI + 量測
    rgb_c, mask, wb_factor = self._process(image_input)
    rel, abs_, med, mask_ok = measurement.measure(rgb_c, mask, wb_factor)
    
    # 2. 品質評估
    quality = measurement.assess_quality(rgb_c, mask)
    quadrant = compute_quadrant_pct(rgb_c, mask)
    
    # 3. spline 推算 pct
    pct = self._compute_prediction(rel, abs_)
    
    # 4. 寫入 DB（每次新增）
    if save_to_db:
        img_id = self.db.insert_prediction_image(rgb_c, filename, account)
        self.db.insert_measurement(img_id, rel, abs_, med, ...)
        self.db.insert_prediction(img_id, self.calibration_id, pct, level, account, ...)
    
    return PredictionResult(...)
```

#### 4.6.3 spline 推算策略 — `_compute_prediction`

```python
pr = clip(self._spline_rel(rel), 0, 110)    # rel-based prediction
pa = clip(self._spline_abs(abs_), 0, 110)   # abs-based prediction

if rel > self._rel_max:           # rel 已超出校準範圍（飽和）
    pct = pa
elif rel > self._rel_max * 1.5:   # 極端高浮黑
    pct = max(95, pa)
else:
    pct = 0.5 * pr + 0.5 * pa     # 平均

return clip(pct, 0, 100)
```

```python
    # 2. Storage.remove(paths)（best effort，失敗不擋 DB）

    # 4. DELETE FROM measurements WHERE image_id IN (...)
    # 5. DELETE FROM images WHERE id IN (...)
    return len(image_ids)
```

刪除順序：predictions → measurements → images，遵守 FK 約束。

## 6. 演算法詳述

### 6.1 ROI 偵測 — Hough Circle Transform

**演算法原理**（Duda & Hart 1972 [4]）：
1. 將像素邊緣（梯度 > param1）視為候選圓周點
2. 對每個候選點，在 (a, b, r) 三維累積空間中投票
3. 累積票數超過 param2 的 (a, b, r) 視為偵測到的圓

**對水泥試體的調整**：
- `dp=1`：累積空間解析度與輸入相同
- `minDist=max(50, min_r)`：避免重複偵測
- `param1=50, param2=30`：適中閾值，平衡靈敏度與誤報
- `minRadius/maxRadius`：限制在影像短邊 18%–48% 之間

**失敗回退**：若無偵測結果，以畫面中心為圓心、預設半徑（30% 短邊）建立 fallback ROI。
不阻擋預測流程，僅在品質警告中標示。

### 6.2 白平衡 — L\* 中位數法

**色彩空間**：CIE 1976 L\*a\*b\* [3]，因為：
- L\* 與人類感知亮度近似線性
- 與 a\*b\* 解耦（色相不影響亮度估計）

**校正公式**：
```
target_L = 95  # CIE 標準白
median_L = median(白紙區的 L*)
wb_factor = target_L / median_L
L*_corrected = L* × wb_factor   (clipped to [0, 100])
```

**設計動機**：避免完整的色彩管理（如 ColorChecker），改用「白紙就在影像中」的快速基準。
此法假設**白紙與試體在同一光源下**，已足以對齊不同手機的曝光差異。

### 6.3 浮黑量化 — Dual Indicator

兩個指標的數學定義（給定 ROI mask M）：

```
rel_dark_pct = | { p ∈ M : L*(p) < median(L*(M)) × rel_threshold } | / |M|

abs_dark_pct = | { p ∈ M : L*(p) < abs_threshold } | / |M|
```

**相關性分析**（基於 mi-20/40/50/70/100 校準資料）：
- rel: target_pct 相關係數 ≈ 0.92（中段最好）
- abs: target_pct 相關係數 ≈ 0.88（高段較線性）

**飽和現象**：
浮黑超過 ~80% 時，rel 中位 L\* 已被浮黑大量拉低，rel\_dark\_pct 不再隨真實 % 增加。
故 v13+ 引入 abs 為輔助指標。

### 6.4 反光偵測（v15）

**簡化策略**：不做去反光，只做警告。
```python
specular_ratio = (L*[roi_mask] > 200).mean()
specular_ok = specular_ratio <= 0.01
```

**與正規方法比較**：
正規去反光（如 Dark Channel Prior, He et al. 2011 [5]）需要計算 dark channel + 反射分量分離，計算複雜。
本系統採用「**SOP 教育替代演算法**」策略，**從拍攝端避免反光**比演算法去反光更有效且零成本。

### 6.5 Smooth Isotonic Regression（校準核心）

詳述見 §4.5.1。完整引文：

> Jiang X, Osl M, Kim J, Ohno-Machado L. **Smooth Isotonic Regression: A New Method to Calibrate Predictive Models.** AMIA Joint Summits on Translational Science Proceedings. 2011; 2011:16-20. PMC3248752. [1]

### 6.6 PCHIP 插值

**Piecewise Cubic Hermite Interpolating Polynomial**（Fritsch & Carlson 1980 [2]）。

**選擇 PCHIP 而非 cubic spline 的原因**：
- PCHIP 保證單調性（給定單調控制點時，輸出曲線也單調）
- 不會像 cubic spline 在資料外側 overshoot
- C¹ 連續（一階導數連續、適合做平滑校準曲線）

實作：`scipy.interpolate.PchipInterpolator`。

### 6.7 LOO MAE 自動評估

```python
def loo_mae(x: ndarray, y: ndarray) -> float:
    errors = []
    for i in range(len(x)):
        mask = np.arange(len(x)) != i
        fitter = smooth_isotonic_fit(x[mask], y[mask])
        if fitter is None: continue
        errors.append(abs(fitter(x[i]) - y[i]))
    return np.mean(errors)
```

用於：
1. 校準時自動選擇最佳 (rel_thr, abs_thr) 組合
2. 給管理員看「這組校準的可信度」
